## 1. 문서 목적

이 문서는 FutureKey의 BRD를 기반으로 MVP 제품 요구사항을 정의하기 위한 PRD 초안이다.

BRD가 비즈니스 방향, 시장 기회, 사용자 가치, 수익 모델을 정의했다면, 이 PRD는 제품팀과 개발팀이 실제 MVP를 설계하고 구현할 수 있도록 다음 항목을 구체화한다.

- MVP의 제품 목표와 비목표
- 핵심 사용자 플로우
- 기능 요구사항 및 우선순위
- 접근 제어, 잠금 해제, 알림, 삭제 등 주요 비즈니스 규칙
- 비기능 요구사항
- 성공 지표 및 출시 단계
- TRD 작성 전 해결해야 할 오픈 질문

## 2. 제품 개요

### 2.1 제품 한 줄 정의

FutureKey는 사용자가 지금 남긴 메시지, 영상 링크, 사진, 음성, 기록을 미래의 특정 시점에 지정된 수신자만 열람할 수 있도록 예약하는 미래 메시지 플랫폼이다.

### 2.2 핵심 제품 약속

> 지금의 마음을, 올바른 사람에게, 올바른 시점에만 열리는 메시지로 남긴다.

### 2.3 핵심 경험

- 발신자는 메시지를 작성하고 미래 공개일을 지정한다.
- 발신자는 수신자에게 키 또는 접근 링크를 공유할 수 있다.
- 수신자는 공개일 전에는 메시지의 존재 또는 최소 메타데이터만 확인할 수 있다.
- 공개일이 되면 수신자는 권한 확인 후 메시지를 열람할 수 있다.
- 메시지는 단순 저장물이 아니라 "언젠가 열릴 예정인 감정적 약속"으로 경험된다.

### 2.4 제품 포지셔닝

FutureKey는 법적 유언장, 상속, 신탁, 금융 집행 서비스가 아니다. MVP는 감정 전달, 미래 예약 메시지, 시간 기반 접근 제어에 집중한다.

## 3. 제품 목표 및 비목표

### 3.1 MVP 목표

1. 사용자가 5분 이내에 첫 미래 메시지를 만들 수 있게 한다.
2. 수신자가 메시지 공개 전에는 내용이 잠겨 있다는 사실을 명확히 이해하게 한다.
3. 지정된 공개일 이후, 권한 있는 수신자가 쉽게 메시지를 열람할 수 있게 한다.
4. "존재 공개 / 내용 비공개" 경험이 기대감과 공유 동기를 만드는지 검증한다.
5. 무료 플랜과 유료 플랜의 전환 포인트를 실험할 수 있는 구조를 만든다.

### 3.2 MVP 비목표

- 법적 유언장 생성, 공증, 상속, 유산 분배
- 공식 가족 관계 검증
- 사망 확인 또는 사망 조건 기반 자동 전달
- 실물 선물 배송, 기프트카드, 기부 집행
- 공개 소셜 피드 또는 커뮤니티 탐색
- 복잡한 기업 광고 상품
- 고위험 금융성 상품 또는 가치 보장 상품
- AI 영상 편집, 자동 전사, 고급 콘텐츠 생성 기능

## 4. 대상 사용자

### 4.1 Primary User: 발신자 / 크리에이터

미래의 누군가에게 메시지를 남기고 싶은 사용자.

대표 사례:
- 부모가 자녀에게 성인이 되는 날 열어볼 메시지를 남김
- 연인이 기념일에 열리는 메시지를 예약
- 친구가 생일, 졸업, 입학 시점에 맞춰 메시지를 남김
- 사용자가 미래의 자기 자신에게 메시지를 남김
- 크리에이터 또는 셀럽이 특정 대상에게 메시지의 존재만 공개

### 4.2 Secondary User: 수신자

지정된 시점에 메시지를 받거나 열람하는 사용자.

대표 사례:
- 자녀, 배우자, 연인, 친구, 가족
- 특정 커뮤니티 멤버
- 팬, 구독자, 지인

### 4.3 Viewer / Audience

메시지 내용은 볼 수 없지만, 발신자가 허용한 범위 내에서 메시지의 존재나 공개 예정일을 볼 수 있는 사용자.

## 5. MVP 범위 요약

### 5.0 우선순위 정의

| 레벨 | 명칭 | 의미 |
|---|---|---|
| P0 | Critical | MVP 출시 차단 요건. 없으면 출시할 수 없다. |
| P1 | High | MVP 완성도 요건. 출시 직전 또는 첫 스프린트 완료 전까지 구현한다. |
| P2 | Medium | 베타 이후 다음 릴리즈에서 구현한다. |
| P3 | Low | 제품-시장 적합성 확인 이후 탐색·구현한다. |

---

### 5.1 MVP 핵심 범위

MVP는 "메시지 생성 -> 예약 -> 잠금 -> 공개일 도달 -> 권한 있는 수신자 열람"의 전체 루프를 검증하는 데 집중한다.

| 영역 | MVP 우선순위 | 설명 |
|---|---:|---|
| 계정 생성 및 로그인 | P0 | Google/Apple 등 OAuth 기반 간편 로그인 |
| 텍스트 메시지 작성 | P0 | 가장 낮은 비용으로 핵심 가치 검증 |
| 외부 영상 링크 등록 | P0 | 무료 플랜 영상 처리 방식 |
| 공개일 예약 | P0 | 날짜/시간 기반 잠금 해제 |
| 키 또는 접근 링크 공유 | P0 | 수신자 접근 권한 부여 |
| 잠긴 메시지 존재 표시 | P0 | 내용은 숨기고 상태만 표시 |
| D-day 카운트다운 | P0 | 기대감 형성 핵심 UX |
| 수신자 열람 | P0 | 공개일 이후 권한 확인 후 콘텐츠 표시 |
| 이메일 알림 | P0 | MVP 기본 알림 채널 |
| 수신 거부 | P0 | 남용 방지 필수 안전 기능 |
| 썸네일 설정 | P1 | 메시지 카드의 감성/공유 품질 향상 |
| 내부 미디어 업로드 | P1 | 유료 플랜 또는 베타 범위 |
| 푸시/SMS 알림 | P1 | 이메일 이후 확장 |
| 다운로드/내보내기 | P1 | 유료 신뢰 기능 |
| 결제/구독 | P1 | MVP 후반 또는 베타 수익화 실험 |
| 기프트카드/실물 선물 | P3 | 제품-시장 적합성 확인 후 탐색 |

### 5.2 MVP에서 제외

- 법적 문서 생성 및 법률 검증
- 사망 여부 인증
- KYC 기반 고위험 인증
- 기업 광고/파트너십 운영 도구
- UGC 템플릿 마켓
- 실물 배송 운영

## 6. 제품 원칙

1. 단순해야 한다: 비기술적 사용자도 안내 없이 첫 메시지를 만들 수 있어야 한다.
2. 프라이버시가 기본값이어야 한다: 공개는 명시적 선택으로만 가능해야 한다.
3. 내용보다 상태가 먼저 보인다: 공개 전에는 내용보다 "언제 열리는지"와 "나에게 남겨졌다"는 경험이 중요하다.
4. 법적 효력을 암시하지 않는다: 모든 UX 문구는 감정 전달 중심이어야 한다.
5. 신뢰를 과장하지 않는다: 장기 보관, 전달 보장, 보안 수준은 실제 구현 가능한 수준으로만 약속한다.
6. 남용을 방지한다: 수신 거부, 신고, 차단, 발신 제한은 MVP부터 포함한다.

## 7. 핵심 사용자 플로우

### 7.1 발신자 첫 메시지 생성 플로우

1. 사용자가 랜딩 또는 앱 진입 화면에서 로그인한다.
2. "새 미래 메시지 만들기"를 선택한다.
3. 메시지 유형을 선택한다: 텍스트 또는 외부 영상 링크.
4. 제목과 본문 또는 영상 링크를 입력한다.
5. 수신자 정보를 입력한다: 이메일 기반으로 시작.
6. 공개일과 시간을 선택한다.
7. 공개 전 수신자에게 존재를 알릴지 선택한다.
8. 공개 범위를 선택한다: 비공개, 링크 공개, 제한 공개.
9. 미리보기를 확인한다.
10. 메시지를 저장하거나 봉인한다.
11. 시스템이 키 또는 접근 링크를 생성한다.
12. 발신자는 링크 공유 또는 이메일 알림 발송을 선택한다.

### 7.2 수신자 공개 전 확인 플로우

1. 수신자가 이메일 또는 링크를 통해 FutureKey에 접속한다.
2. 수신자는 OAuth로 로그인한다.
3. 시스템이 수신자 권한과 접근 링크 유효성을 확인한다.
4. 공개일 전이면 메시지 내용은 표시하지 않는다.
5. 허용된 경우 다음 정보만 표시한다:
   - 메시지가 존재한다는 사실
   - 공개 예정일
   - 카운트다운
   - 발신자가 공개를 허용한 최소 메타데이터
6. 수신자는 알림 수신 설정 또는 수신 거부를 선택할 수 있다.

### 7.3 수신자 공개일 이후 열람 플로우

1. 공개일이 되면 시스템이 수신자에게 이메일 알림을 보낸다.
2. 수신자가 링크를 열고 로그인한다.
3. 시스템이 권한, 공개일, 링크 만료 여부, 수신 거부 상태를 확인한다.
4. 조건을 모두 만족하면 메시지 내용을 표시한다.
5. 수신자는 메시지를 읽거나 외부 영상을 열람한다.
6. 유료 기능이 활성화된 경우 수신자는 콘텐츠를 다운로드하거나 보관할 수 있다.

### 7.4 공개 페이지 플로우

1. 발신자가 메시지 존재 공개를 선택한다.
2. 공개 페이지 방문자는 메시지의 존재와 공개 예정일을 볼 수 있다.
3. 공개 전에는 내용, 상세 발신자 정보, 수신자 개인정보를 표시하지 않는다.
4. 공개일 이후에도 실제 내용은 권한 있는 사용자에게만 표시한다.

## 8. 기능 요구사항

### 8.1 계정 및 인증

| ID | 요구사항 | 우선순위 | 수용 기준 |
|---|---|---:|---|
| AUTH-001 | 사용자는 OAuth 기반으로 가입 및 로그인할 수 있어야 한다. | P0 | Google 로그인을 통해 계정 생성과 재로그인이 가능하다. |
| AUTH-002 | 사용자는 기본 프로필을 관리할 수 있어야 한다. | P0 | 이름, 이메일, 프로필 표시명을 확인하고 수정할 수 있다. |
| AUTH-003 | 수신자도 메시지 열람 전 인증되어야 한다. | P0 | 수신자는 링크만으로 내용에 접근할 수 없고 로그인 후 권한 검증을 통과해야 한다. |
| AUTH-004 | 계정 복구 흐름이 제공되어야 한다. | P1 | 사용자가 로그인 제공자 접근 실패 시 지원 요청 또는 대체 이메일 확인을 진행할 수 있다. |

### 8.2 메시지 생성

| ID | 요구사항 | 우선순위 | 수용 기준 |
|---|---|---:|---|
| MSG-001 | 사용자는 텍스트 메시지를 작성할 수 있어야 한다. | P0 | 제목과 본문을 입력하고 저장할 수 있다. |
| MSG-002 | 사용자는 외부 영상 링크를 등록할 수 있어야 한다. | P0 | YouTube 등 외부 링크를 입력하고 메시지에 연결할 수 있다. |
| MSG-003 | 시스템은 외부 링크 형식을 검증해야 한다. | P0 | 잘못된 URL은 저장되지 않고 이해하기 쉬운 오류가 표시된다. |
| MSG-004 | 사용자는 메시지에 선택적 설명을 추가할 수 있어야 한다. | P2 | 설명 필드는 비워도 저장 가능하다. |
| MSG-005 | 사용자는 사진, 음성, 영상 파일을 직접 업로드할 수 있어야 한다. | P1 | 유료 또는 베타 사용자에게 내부 저장 업로드가 제공된다. |
| MSG-006 | 사용자는 메시지 썸네일을 설정할 수 있어야 한다. | P1 | 이미지, 텍스트 카드, 기본 템플릿 중 하나가 카드 대표 이미지로 표시된다. |

### 8.3 예약 및 시간 잠금

| ID | 요구사항 | 우선순위 | 수용 기준 |
|---|---|---:|---|
| LOCK-001 | 사용자는 메시지 공개 날짜와 시간을 지정할 수 있어야 한다. | P0 | 현재 시점 이후 날짜/시간을 저장할 수 있다. |
| LOCK-002 | 공개일 전에는 메시지 본문과 미디어에 접근할 수 없어야 한다. | P0 | 권한 있는 수신자도 공개일 전에는 내용 대신 잠김 상태를 본다. |
| LOCK-003 | 공개일 도달 후 권한 있는 수신자만 내용을 볼 수 있어야 한다. | P0 | 권한, 시간, 링크 유효성이 모두 충족될 때만 내용이 표시된다. |
| LOCK-004 | 시스템은 사용자 현지 시간대를 명확히 표시해야 한다. | P0 | 예약 화면과 수신자 화면에 날짜/시간대가 일관되게 표시된다. |
| LOCK-005 | 무료 플랜은 장기 예약 기간 제한을 적용할 수 있어야 한다. | P1 | 무료 사용자는 설정된 최대 예약 기간을 넘는 날짜 선택 시 업그레이드 안내를 본다. |

### 8.4 봉인 및 수정 정책

| ID | 요구사항 | 우선순위 | 수용 기준 |
|---|---|---:|---|
| SEAL-001 | 메시지는 초안 상태로 저장될 수 있어야 한다. | P0 | 초안은 발신자만 볼 수 있고 수신자에게 알림이 가지 않는다. |
| SEAL-002 | 발신자는 메시지를 봉인할 수 있어야 한다. | P0 | 봉인 후 메시지는 예약 활성 상태가 되고 접근 링크 또는 알림을 생성할 수 있다. |

## 9. 주요 비즈니스 규칙

### 9.1 메시지 상태

| 상태 | 의미 |
|---|---|
| Draft | 작성 중이며 발신자만 접근 가능 |
| Sealed | 발신자가 확정했으며 예약/공유 가능 |
| Locked | 공개일 전이라 수신자가 내용 열람 불가 |
| Available | 공개일이 지나 권한 있는 수신자가 열람 가능 |
| Expired | 링크 또는 보관 기간이 만료됨 |
| Deleted | 사용자 화면에서 삭제됨 |
| Soft Deleted | 내부 보존 정책에 따라 복구 가능 상태로 보관됨 |

### 9.2 공개 전 메타데이터

공개 전 기본 노출 정보는 최소화한다.

기본값:
- 메시지 존재 여부
- 공개 예정일
- 카운트다운

발신자가 명시적으로 허용한 경우에만 추가 노출 가능:
- 제목
- 썸네일
- 발신자 표시명
- 짧은 설명

공개 전에는 절대 노출하지 않는 정보:
- 메시지 본문
- 영상/음성/사진 파일
- 외부 영상 링크 원본 URL
- 수신자 개인정보
- 다른 수신자 목록

### 9.3 공개일 및 시간대

- 공개일은 발신자가 설정한 시간대 기준으로 저장한다.
- 수신자 화면에는 수신자 현지 시간대 변환을 함께 표시한다.
- 서버는 UTC 기준으로 공개 가능 여부를 판정한다.
- 공개일 도달 시 즉시 열람 가능해야 하지만, 알림 발송 지연이 열람 자체를 막아서는 안 된다.

### 9.4 수신 거부

- 수신자는 특정 메시지, 특정 발신자, 또는 플랫폼 전체 알림을 거부할 수 있다.
- 수신 거부 시 신규 알림은 발송되지 않는다.
- 발신자에게는 구체적 거부 사유를 표시하지 않는다.
- 반복 차단된 발신자는 운영 검토 대상이 된다.

### 9.5 공유 링크 만료

- 공유 링크는 기본적으로 만료일을 가진다.
- MVP 기본 만료 기간은 제품/보안 논의 후 확정한다.
- 만료된 링크는 메시지 내용이나 개인정보를 노출하지 않는 안내 페이지로 이동한다.

### 9.6 무료/유료 제한

초기 값은 제품 실험을 위해 설정 가능해야 하며, PRD 확정 전까지 다음 수치는 TBD로 둔다.

- 무료 월간 메시지 생성 한도
- 무료 예약 가능 최대 기간
- 무료 외부 영상 링크 개수
- 유료 내부 저장 용량
- 유료 장기 예약 가능 기간
- 다운로드/내보내기 제공 범위

## 11. UX 요구사항

### 11.1 톤 앤 매너

제품은 따뜻하고, 차분하고, 신뢰할 수 있어야 한다. 죽음, 법적 효력, 상속보다 미래 메시지, 기억, 마음, 기대감 중심의 언어를 사용한다.

### 11.2 온보딩

MVP는 첫 사용자가 핵심 개념을 빠르게 이해하도록 다음을 제공한다.

- 짧은 3단계 온보딩: 남기기, 잠그기, 열리는 날 받기
- 첫 메시지 작성 CTA
- 카테고리별 템플릿 제안
- 샘플 잠긴 메시지 미리보기
- 스킵 가능한 구조

### 11.3 메시지 템플릿

템플릿은 작성 장벽을 낮추는 핵심 UX로 간주한다.

초기 템플릿 예시:
- 생일/기념일
- 부모에서 자녀에게
- 연인/부부
- 친구에게
- 미래의 나에게
- 졸업/입학/새 출발

무료 플랜은 제한된 기본 템플릿을 제공하고, 유료 플랜은 확장 템플릿 라이브러리로 수익화할 수 있다.

### 11.4 접근성

- WCAG 2.1 AA 준수를 목표로 한다.
- 키보드만으로 핵심 플로우를 완료할 수 있어야 한다.
- 주요 버튼과 상태에는 명확한 텍스트 레이블을 제공한다.
- 카운트다운은 색상만으로 상태를 전달하지 않는다.
- 고령자 사용자를 고려해 글자 크기, 대비, 오류 메시지를 명확하게 설계한다.

## 12. 비기능 요구사항

### 12.1 보안 및 프라이버시

- 저장 데이터와 전송 데이터는 암호화한다.
- 접근 제어는 서버에서 검증한다.
- 외부 링크 원본은 공개 전 수신자 또는 공개 페이지에 노출하지 않는다.
- 민감한 액션은 감사 로그를 남긴다.
- 기본 공개 설정은 비공개여야 한다.

### 12.2 신뢰성

- 공개일 도달 후 권한 있는 수신자의 열람 성공률을 핵심 품질 지표로 추적한다.
- 이메일 발송 실패 시 재시도 또는 대체 안내를 제공한다.
- 장기 예약 메시지는 백업 및 복구 정책이 필요하다.

### 12.3 성능

- 메시지 목록은 일반 네트워크 환경에서 빠르게 로드되어야 한다.
- 카운트다운 UI는 사용자의 기기 시간을 신뢰하지 않고 서버 공개 가능 상태를 최종 기준으로 삼는다.
- 외부 영상 링크 미리보기 실패가 메시지 저장을 막아서는 안 된다.

### 12.4 모바일

- 모바일 웹에서 메시지 작성, 링크 등록, 공개일 선택, 수신자 열람이 가능해야 한다.
- 영상 링크 등록과 카운트다운 확인은 모바일 우선으로 설계한다.

### 12.5 법무 및 컴플라이언스

- 제품 문구는 법적 유언, 상속, 재산 이전 효력을 암시하지 않는다.
- 개인정보 삭제, 보존, 수신 거부 정책은 서비스 약관과 개인정보 처리방침에 반영되어야 한다.
- KYC는 MVP 범위 밖이지만, 선물/기부/고액 거래 확장 시 재검토한다.

## 13. 성공 지표

### 13.1 활성화 지표

- 가입 후 첫 메시지 생성율
- 첫 메시지 생성까지 걸린 시간
- 온보딩 완료율
- 메시지 봉인 완료율

### 13.2 핵심 제품 지표

- 예약 메시지 생성 수
- 공개일이 설정된 메시지 비율
- 수신자 접근 링크 활성화율
- 공개일 이후 열람 완료율
- 카운트다운 페이지 재방문율

### 13.3 성장 지표

- 공유 링크 생성 비율
- 공개 페이지 조회수
- 수신자에서 가입자로 전환율
- 초대 또는 공유 기반 신규 가입 비중

### 13.4 수익화 지표

- 무료 한도 도달 사용자 비율
- 무료에서 유료 전환율
- 유료 사용자의 장기 예약 생성 비율
- 내부 저장 기능 사용률
- 다운로드/내보내기 사용률

### 13.5 신뢰 및 안전 지표

- 전달 성공률
- 이메일 발송 실패율
- 권한 실패율
- 신고/차단 비율
- 사용자 신뢰 관련 CS 비율

## 15. 주요 리스크 및 대응

| 리스크 | 설명 | MVP 대응 |
|---|---|---|
| 법적 유언 서비스로 오해 | 사용자가 상속/유언 기능으로 오해할 수 있음 | 문구와 약관에서 감정 메시지 플랫폼임을 명확히 표현 |
| 장기 신뢰 부족 | 사용자가 "몇 년 뒤에도 전달될까"를 걱정 | 초기에는 짧은 예약 기간과 명확한 보관 정책으로 시작 |
| 프라이버시 침해 | 민감한 메시지 또는 링크 노출 가능 | 기본 비공개, 서버 권한 검증, 링크 만료, 감사 로그 |
| 원치 않는 메시지 | 스토킹성 또는 괴롭힘 메시지 가능 | 수신 거부, 차단, 신고, 반복 차단 발신자 제한 |
| 외부 링크 만료 | 무료 영상 링크가 삭제되거나 비공개 변경될 수 있음 | 무료 플랜 한계 안내, 유료 내부 저장 옵션 제공 |
| 상업화 반감 | 감정적 순간에 광고/결제가 과하게 느껴질 수 있음 | MVP는 구독/저장 가치 중심, 광고는 후순위 |

## 16. 오픈 질문

다음 항목은 TRD 또는 디자인 상세화 전 결정이 필요하다.

1. 무료 플랜의 메시지 생성 한도는 월간 기준인가, 전체 누적 기준인가?
2. 무료 플랜의 최대 예약 기간은 1년이 적절한가?
3. 수신자는 공개 전 발신자 정보를 볼 수 있어야 하는가?
4. 공개 전 제목과 썸네일은 기본 비공개인가, 발신자 선택 공개인가?
5. 메시지 봉인 후 수정 정책은 완전 잠금인가, 재봉인 가능한 수정인가?
6. 접근 키는 계정 기반 권한인가, 만료 링크인가, 둘의 조합인가?
7. 공개일 이후 링크 만료 기본 기간은 30일, 90일, 또는 무기한 중 무엇인가?
8. 삭제된 메시지의 소프트 딜리트 보존 기간은 1년, 2년, 또는 플랜별 차등인가?
9. 내부 미디어 업로드는 Public MVP에 포함할 것인가, 유료 베타로 둘 것인가?
10. 결제는 MVP 출시 전 포함할 것인가, 수동 베타 판매 이후 도입할 것인가?
11. 메시지 템플릿은 제품 핵심 기능인가, 유료 전환 기능인가?
12. 신고된 메시지를 운영자가 어느 수준까지 열람할 수 있어야 하는가?

## 17. TRD 작성 시 전달 사항

TRD에서는 다음 기술 영역을 우선 구체화해야 한다.

- 인증 및 세션 구조
- 시간대 저장 및 공개 가능 여부 판정 방식
- 접근 링크/키 설계
- 메시지 상태 머신
- 알림 스케줄러 및 재시도 정책
- 외부 영상 링크 저장 및 검증 방식
- 미디어 업로드와 스토리지 수명주기
- 서버 기반 접근 제어
- 감사 로그 설계
- 소프트 딜리트 및 데이터 보존 정책
- 플랜 권한 및 기능 플래그 구조

## Appendix A. 용어 정의

| 용어 | 정의 |
|---|---|
| 발신자 / 크리에이터 | 미래 메시지를 만드는 사용자 |
| 수신자 | 메시지를 받을 사용자 |
| Viewer / Audience | 내용은 못 보지만 존재 또는 공개 예정 정보를 볼 수 있는 사용자 |
| 메시지 | 텍스트, 외부 영상 링크, 향후 미디어 파일을 포함하는 예약 콘텐츠 |
| 키 | 특정 메시지 또는 볼트에 접근할 수 있는 권한 객체 |
| 접근 링크 | 수신자 또는 공개 페이지 진입을 위한 만료 가능한 링크 |
| 봉인 | 메시지를 확정하여 예약/공유 가능한 상태로 만드는 행위 |
| 공개일 / D-day | 메시지 잠금이 해제되는 날짜와 시간 |
| 존재 공개 | 메시지 내용 없이 "메시지가 있다"는 사실만 보여주는 설정 |
| 내부 저장 | FutureKey 플랫폼이 직접 미디어 파일을 보관하는 유료 중심 기능 |
